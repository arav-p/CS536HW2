\section{Introduction}

This report presents the implementation and analysis of a custom iPerf3-compatible TCP client for measuring network throughput and TCP congestion control behavior. The assignment requires building a socket-based client from scratch, extracting TCP statistics during transfer, and training an ML model to predict congestion window updates. All experiments were conducted using Docker containers on Ubuntu 24.04 to ensure Linux kernel compatibility for TCP\_INFO access.

\textbf{Note:} The quantitative results and observations in this report are based on data collected from AWS EC2 Ubuntu instances. Due to instance termination, the visualization figures are described textually rather than included as images. The analysis is based on real Linux TCP statistics collected during the experiments.

The implementation consists of three main components:
\begin{enumerate}
    \item A Python TCP client compatible with public iPerf3 servers
    \item Real-time TCP statistics extraction and visualization
    \item Machine learning model for congestion window prediction
\end{enumerate}

\section{Methodology}

\subsection{iPerf Client Implementation}

The client was implemented in Python using the socket library, following the iPerf3 protocol specification. Key components include:

\begin{itemize}
    \item Control connection establishment with JSON parameter exchange
    \item Data connection for continuous transmission
    \item Robust error handling for server failures and timeouts
    \item Server discovery from public iPerf3 server lists
\end{itemize}

\subsection{TCP Statistics Extraction}

TCP statistics were extracted using Python's socket.getsockopt() with TCP\_INFO, capturing:
\begin{itemize}
    \item Congestion window (snd\_cwnd)
    \item Round-trip time estimates (srtt, rttvar)
    \item Loss signals (retransmits, lost packets)
    \item Throughput measurements
\end{itemize}

Code location: \href{https://github.com/arav-p/CS536HW2/blob/main/tcp_stats.py}{tcp\_stats.py} (\url{https://github.com/arav-p/CS536HW2/blob/main/tcp_stats.py})

\subsection{ML Model Training}

A linear regression model was trained to predict congestion window changes ($\Delta$cwnd). Sample weights implement the objective function with a corrected off-by-one shift: weight at time $t-1$ is derived from the outcome $\eta$ at time $t$, matching the spec definition $\eta(t-1) = goodput(t) - \alpha \cdot RTT(t) - \beta \cdot loss(t)$.

\begin{itemize}
    \item Features: goodput, RTT, loss rate, lagged values, moving averages
    \item Target: $\Delta$snd\_cwnd
    \item Objective: $\eta = goodput - \alpha \cdot RTT - \beta \cdot loss$
\end{itemize}

Code location: \href{https://github.com/arav-p/CS536HW2/blob/main/ml_model.py}{ml\_model.py} (\url{https://github.com/arav-p/CS536HW2/blob/main/ml_model.py})

\subsection{Experimental Setup}

Experiments were conducted on AWS EC2 (Ubuntu 24.04) to ensure Linux kernel compatibility. The Docker container standardizes the user-space environment while leveraging the host kernel for TCP statistics.

\subsubsection{Dataset Construction}

Features used:

feature_cols = [

\textbf{Note on feature exclusion:} \texttt{snd\_cwnd} is excluded from the feature set. The target $\Delta$cwnd is defined as \texttt{snd\_cwnd} $-$ \texttt{cwnd\_lag1}; including \texttt{snd\_cwnd} would allow the model to trivially reconstruct the target via an identity relationship (R\textsuperscript{2}=1.0), masking any genuine predictive signal. With \texttt{snd\_cwnd} removed, the model must learn from real TCP congestion signals.

The dominant signal is the near-equal and opposite pair \texttt{rtt\_ma3} ($-1487.89$) and \texttt{rtt\_ms} ($+1448.63$). Their net effect is $\approx$ rtt\_ms $-$ rtt\_ma3 — the deviation of current RTT from its 3-sample moving average — which acts as an implicit RTT derivative. A rising RTT above its recent average signals queue build-up and causes $\Delta$cwnd to decrease, while RTT falling below its average signals relief and allows growth. \texttt{cwnd\_lag1} ($-318.75$) captures the autoregressive inertia of the congestion window.

Their net effect computes RTT deviation from its moving average, acting as an implicit RTT derivative and detecting queue build-up before loss occurs.
    \item \textbf{Data Statistics (from EC2 run):}

        \item cwnd: median = 3,984 pkts, p75 = 3,988 pkts
        \item Goodput: median = 220\,Mbps

    \item \textbf{Correlation with Goodput:}
    \begin{itemize}

        \item RTT $\leftrightarrow$ goodput: $-0.766$ (strong negative — RTT is the dominant congestion signal)

    \end{itemize}
\end{enumerate}

\textbf{EXTRACTED ALGORITHM (derived dynamically from model output):}

The algorithm below is generated automatically by \texttt{extract\_algorithm()} in \texttt{ml\_model.py}. Parameters (RTT thresholds, BDP target, loss threshold, congestion avoidance coefficients) are derived from the actual learned linear coefficients, data percentiles, and bandwidth-delay product computed from each run's measurements.

\begin{lstlisting}
Simplified Hand-Written Algorithm (derived from model observations)
-------------------------------------------------------------------
  queue_delay = rtt - rtt_min    # isolate queueing component

  if loss > 0.0:
    cwnd = max(2, cwnd * 0.5)               # AIMD multiplicative decrease

  elif queue_delay > (rtt_min * 13.09):
    cwnd = max(2, cwnd - 1)                 # Vegas-like delay decrease

  elif cwnd < 1769.9:
    cwnd = cwnd + 1                         # probe toward BDP target

  else:
    delta  = -1487.88637 * rtt_ma3
    delta +=  1448.63302 * rtt_ms
    delta +=  -318.74656 * cwnd_lag1
    cwnd = max(2, cwnd + delta)             # RTT-derivative congestion avoidance

  cwnd = max(2, min(cwnd, 65535))

Grounding:
  BDP      = 1769.9 pkts  (goodput x RTT / MSS=1448B)
  RTT_min  = 7.51 ms      (propagation baseline, 5th percentile)
  Dominant signal: rtt_ma3 = -1487.89  (RTT deviation from moving average)
\end{lstlisting}

\textbf{Key Principles:}
\begin{itemize}
    \item Loss $\Rightarrow$ AIMD multiplicative decrease (halve cwnd)
    \item Delay above threshold $\Rightarrow$ Vegas-like gentle decrease (queue signal before loss)
    \item Below BDP target $\Rightarrow$ additive increase to probe for bandwidth
    \item At steady state $\Rightarrow$ RTT-derivative term adjusts cwnd based on whether RTT is rising or falling relative to its moving average
    \item Hard bounds: cwnd $\in [2, 65535]$
\end{itemize}

This algorithm is grounded entirely in the data: thresholds (BDP = 1769.9 pkts, RTT\textsubscript{min} = 7.51\,ms) and coefficients ($-1487.89$, $+1448.63$, $-318.75$) are derived automatically from the learned linear model and observed data percentiles.

\section{Discussion}

\subsection{Technical Challenges}

1. \textbf{iPerf3 Protocol Implementation:} Reverse-engineering the JSON-based handshake required careful packet analysis and multiple iterations to handle server variations.

2. \textbf{TCP Statistics Access:} macOS limitations necessitated EC2 deployment for reliable TCP\_INFO extraction.

3. \textbf{ML Data Leakage (resolved):} The target $\Delta$cwnd is defined as \texttt{snd\_cwnd} $-$ \texttt{cwnd\_lag1}. Including \texttt{snd\_cwnd} as a feature creates an identity relationship that trivially yields R\textsuperscript{2}=1.0. We resolved this by removing \texttt{snd\_cwnd} from the feature set. The resulting model (Train R\textsuperscript{2}=0.439, Test cwnd sequence R\textsuperscript{2}=0.955) learns genuine TCP signals: the dominant coefficients on \texttt{rtt\_ms} and \texttt{rtt\_ma3} capture RTT deviation from moving average as an implicit congestion derivative.

\subsection{Limitations}

1. \textbf{Platform Dependence:} TCP behavior varies across kernels; results are specific to Linux TCP implementation.

2. \textbf{Server Variability:} Public iPerf servers have different capacities and network conditions.

3. \textbf{Real-time Constraints:} Socket option extraction adds minimal latency but may affect high-frequency measurements.

\section{Conclusion}

This implementation successfully demonstrates end-to-end TCP congestion control analysis, from protocol-level client implementation through statistical analysis to ML-based algorithm extraction. The results provide insights into real-world TCP behavior and validate the effectiveness of AIMD-based congestion control.

Key achievements:
\begin{itemize}
    \item Robust iPerf3-compatible client with automatic server fallback
    \item Real-time TCP statistics extraction on Linux
    \item ML model for congestion window prediction
    \item Data-grounded congestion control algorithm
\end{itemize}

The Docker-based approach ensures reproducibility across Linux environments, making the analysis portable and verifiable. A \texttt{run\_experiment.sh} script provides single-command execution: it builds the image and runs the container with \texttt{--network host} and volume mounts so results persist to \texttt{./data} and \texttt{./plots} on the host without manual file extraction.
